## Summary
Successfully implemented full-text search using PostgreSQL tsvector with GIN index for the Memory Stream Go backend.

## Deliverables

### 1. Database Migration
- File: `go-server/migration/sql/010_search_vector.sql`
- Added `search_vector` tsvector generated stored column to cards table
- Created GIN index `idx_cards_search_vector` for fast full-text search
- Uses 'simple' config for CJK support

- Column definition:
  ```sql
  search_vector tsvector
    GENERATED ALWAYS AS (
      setweight(to_tsvector('simple', coalesce(title, '')), 'A') ||
      setweight(to_tsvector('simple', coalesce(raw_md, '')), 'B')
    ) STORED
  ```
- Index creation
  ```sql
  CREATE INDEX idx_cards_search_vector ON cards USING GIN(search_vector);
  ```

### 2. Service Layer
- File: `go-server/internal/services/search.go`
- `SearchService` struct with database connection
- `SearchCards(query string, limit, offset int)` method
- Returns `SearchResult` slice with ID, title, excerpt, and rank
- Input validation: empty query returns error
- Pagination: limit defaults to 20, capped at 100
- Uses `plainto_tsquery('simple', ?)` for safe user input handling
- Uses `ts_rank()` for relevance ordering
- Returns total count and results

### 3. Handler Layer

  - `limit` (optional): max results per page (default: 20, max: 100)

## Technical Details

- **tsvector Configuration**: `simple`
  - Reason: Project contains Chinese content
  - `simple` config tokenizes text without stemming
  - Better CJK support compared to `english` config

- **Generated Stored Column**
  - `search_vector` is auto-computed by PostgreSQL
  - Combines weighted title (weight 'A') and content (weight 'B')
  - No triggers needed - PostgreSQL handles updates automatically
  - Column formula:
    ```sql
    setweight(to_tsvector('simple', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('simple', coalesce(raw_md, '')), 'B')
    ```

- **Search Query**
  - Uses `plainto_tsquery('simple', ?)` for safe user input
  - Uses `ts_rank()` for relevance scoring
  - Results ordered by rank descending
  - Separate count query for total matching cards

- **API Design**
  - Public endpoint (no auth required)
  - Follows existing handler patterns
  - Consistent error handling using `appErr` package
  - JSON response with results, total, and original query
